ASCON Hardware Accelerator / Embedded Secure Boot

Hardware/security project focused on extending a RISC-V Ibex-based system with an ASCON cryptographic accelerator and using it for embedded secure boot verification.

Course: Hardware Design for Security — TU Graz
Project: Embedded Secure Boot with ASCON acceleration
Languages: SystemVerilog, C, Python
Main topics: ASCON, secure boot, RISC-V Ibex, custom instructions, hardware/software co-design

Overview

This project focused on adding modern security functionality to an embedded CPU system. The main idea was to accelerate ASCON cryptographic operations in hardware and use the accelerator from software to verify boot images.

Secure boot verifies the integrity and authenticity of a software image before execution. If verification fails, the bootloader refuses to boot the image. Because embedded devices have limited processing power, this project used hardware acceleration to make cryptographic operations more suitable for the target system.

What I Worked On

  • Integrated an ASCON accelerator into a RISC-V Ibex-based system.
  • Implemented a hardware wrapper around the ASCON permutation module.
  • Maintained a 320-bit ASCON internal state inside the accelerator.
  • Added custom instruction support for ASCON operations.
  • Worked with AXOR for state-word XOR operations.
  • Worked with APERM for multi-round ASCON permutation execution.
  • Modified decode and execute-stage logic for ASCON instruction handling.
  • Implemented stall/control behavior for multi-cycle accelerator operations.
  • Implemented ASCON-Hash256 and ASCON-AEAD128 usage from software.
  • Worked on secure boot image packing and bootloader verification flow.

Technical Details

ASCON Accelerator Architecture

The accelerator was split into two main hardware modules: a combinational ASCON permutation module and an accelerator wrapper connected to the CPU. The wrapper stores the ASCON internal state, starts AXOR/APERM operations, generates control signals, and routes results back to the processor.

Internal State Representation

The 320-bit ASCON state was represented as ten 32-bit words. This made it convenient to access individual state words through immediate values in the custom instruction format.

Custom Instructions

Two custom instructions were used:

  • AXOR — XORs a CPU register value with a selected ASCON state word and stores the result back into the accelerator state.
  • APERM — performs a defined number of ASCON permutation rounds on the internal accelerator state.

Pipeline Integration

The decoder was extended to recognize ASCON instructions by opcode and function fields. The execute stage instantiated the accelerator and selected ASCON results through the result multiplexer. Multi-cycle APERM operations required busy/valid feedback from execute back to decode so that the pipeline could stall correctly until the operation completed.

Cryptographic Software

On the software side, ASCON-Hash256 was used to compute a 256-bit digest of the boot image. ASCON-AEAD128 was used to compute an authentication tag. The implementation used the hardware accelerator through the custom instructions instead of doing the full permutation purely in software.

Secure Boot Image Format

The boot image contained a magic header, image length, ELF binary, ASCON hash digest, and ASCON-AEAD authentication tag. During boot, the bootloader recomputed the hash and tag using the hardware implementation and compared them against the stored values to verify integrity and authenticity.

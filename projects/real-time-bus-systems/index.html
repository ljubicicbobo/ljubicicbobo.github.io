Real-Time Bus Systems Lab

Practical CAN bus project focused on communication analysis, physical-layer signal inspection, CAN frame encoding/decoding, CRC handling, bit stuffing, transceiver behavior, and arbitration testing on real embedded hardware.

Course: Real-Time Bus Systems Lab — TU Graz
Platform: TI MSP430F5529 LaunchPad, B&R PLC CAN hardware
Language: C
Main topics: CAN bus, arbitration, CRC, bit stuffing, oscilloscope analysis

Overview

This lab focused on understanding CAN communication from both the software and physical-layer side. The exercises included sending and receiving CAN messages, analyzing packets with an oscilloscope, building and testing a CAN transceiver, encoding and decoding CAN frames, and testing a multi-node CAN network.

The lab used B&R PLC stations for early CAN communication analysis and a TI MSP430F5529 LaunchPad for implementing and testing CAN frame handling on embedded hardware.

What I Worked On

  • Configured and analyzed CAN communication between hardware nodes.
  • Used an oscilloscope to inspect CAN physical-layer signals.
  • Decoded CAN frames into fields such as arbitration, control, data, CRC, ACK, and end-of-frame.
  • Implemented CAN frame creation and transmission logic in C.
  • Worked with bit stuffing and de-stuffing rules.
  • Implemented and checked CRC handling for CAN frames.
  • Tested CAN arbitration behavior between competing client nodes.
  • Analyzed standard vs. extended CAN frame priority behavior.
  • Investigated termination resistor effects and bus-fault behavior.

Technical Details

CAN Communication and Physical-Layer Analysis

In the first part of the lab, we worked with CAN communication on B&R PLC hardware and used an oscilloscope to capture CAN-High and CAN-Low signals. This helped connect the abstract CAN frame format to the actual differential bus signals.

CAN Frame Encoding and Decoding

The software part involved implementing CAN frame creation, transmission, reception, and decoding. The implementation had to handle standard and extended identifiers, payload length, data fields, CRC calculation, ACK behavior, and error handling.

Bit Stuffing and CRC

A key part of the project was understanding how CAN inserts stuffing bits after long runs of equal bits and how the receiver removes them again. CRC handling was implemented using a shift-register-style approach to validate received frames.

CAN Arbitration Test

In the network test, multiple nodes competed for the CAN bus. We defined operation IDs for addition, subtraction, and multiplication requests and verified arbitration behavior using oscilloscope captures. The test showed how lower-priority frames stop transmitting when they detect that another node is driving a dominant bit on the bus.

Network Setup

The final setup used a server node and two client nodes. The server provided addition, subtraction, and multiplication services. One client sent standard CAN frames for addition/subtraction requests, while another client sent an extended CAN frame for multiplication requests.
